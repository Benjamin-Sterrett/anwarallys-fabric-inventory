# Debug Report — PRJ-878

## Problem Statement

In `src/routes/folder.tsx`, the Retry button on error state only re-subscribes the child folder listener. The two `getFolderById` fetches (current folder metadata + ancestor breadcrumb labels) run in separate effects with dependency arrays that do NOT include the retry counter. When Retry is pressed, only the child subscription reruns; metadata fetches stay stale.

## IS / IS NOT

| Dimension | IS | IS NOT |
|-----------|-----|--------|
| What | Retry button reruns child subscription only | Retry does NOT rerun folder metadata or breadcrumb fetches |
| Where | `src/routes/folder.tsx` effects | Not in child subscription logic (that works) |

## Root Cause

- **ROOT-CAUSE-PROVEN: true**
- The metadata fetch effects depend on `folderId` only, not on a shared retry nonce. The Retry button increments a counter that only the child subscription effect observes.
- Proximate cause: missing retry state in metadata effect dependency arrays.
- Root cause: three separate retry mechanisms instead of one unified nonce.

## Fix Approach

Replace separate retry flags with a single `retryNonce` state. Add `retryNonce` to dependency arrays of:
1. Current folder metadata fetch effect
2. Ancestor breadcrumb fetch effect  
3. Child subscription effect (already has something similar — unify)

Expected: ~10 LOC in `src/routes/folder.tsx`.

## Verification Plan

1. `pnpm typecheck && pnpm build` pass.
2. Manual smoke: throttle to Offline, navigate to `/folders/SOME_ID`, click Retry after restore — title, breadcrumb, children all populate.
