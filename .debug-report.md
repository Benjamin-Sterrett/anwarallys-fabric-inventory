## Observations

### Observation 1 — code path
**File/lines:** `src/lib/queries/items.ts:117-134`
```ts
export async function listActiveItemsInFolder(folderId: string): Promise<Result<RollItem[]>> {
  ...
  const q = query(
    collection(db, 'items').withConverter(itemConverter),
    where('folderId', '==', folderId),
    where('deletedAt', '==', null),
    orderBy('sku'),
  );
  return ok((await getDocs(q)).docs.map((d) => d.data()));
```
**Interpretation:** One-shot `getDocs` (not `onSnapshot`). Equality on `folderId` and `deletedAt`, order by `sku`. The error code returned is exactly `firestore/permission-denied` per the boundary's `FirebaseError` mapping at line 131.

### Observation 2 — rule asymmetry vs symptom asymmetry
The items list rule and the folders list rule are textually identical (`if isActiveStaff()`), yet items list fails while folders list succeeds for the same authenticated user in the same view. This is the central diagnostic puzzle.

### Observation 3 — AuthBar render evidence
The AuthBar shows "Signed in as Shaaiz". The displayName comes from `/users/{auth.uid}.displayName` (per PRJ-876 Firestore-source-of-truth fix). Reading `/users/{uid}` is gated by:
```
allow get: if isAdminUser() || (isActiveStaff() && uid == request.auth.uid);
```
For this read to succeed, EITHER `isAdminUser()` OR `isActiveStaff()` must be true. Since the items list (which uses `isActiveStaff`) fails, AuthBar is succeeding via the `isAdminUser` branch. **`isAdminUser()` is therefore passing for Shaaiz; `isActiveStaff()` is failing.**

### Observation 4 — `isAdminUser` vs `isActiveStaff` decomposition
- `isAdminUser` requires: `isAuthed() && email_verified == true && email.lower() == /config/admin.adminEmail.lower()`
- `isActiveStaff` requires: `isAuthed() && exists(/users/{uid}) && get(/users/{uid}).data.isActive == true`

Both require `isAuthed()` (passes — user is signed in). The difference is the second/third condition. Since `isActiveStaff` is failing AND AuthBar's admin-branch read on `/users/{uid}` succeeded (which means `/users/{uid}` exists), the failure is at `data.isActive == true`.

## Root Cause

- **ROOT-CAUSE-PROVEN: true**
- **Statement:** The Firestore Security Rules in the repo (PRJ-805 + PRJ-865/866 + PRJ-873/874/877) were never deployed to Firebase. The live rules in production are the pre-PRJ-805 prototype rules from 2026-04-18, which contain no `/items/` block (defaults to deny) and a permissive `match /folders/{folderId} { allow read, write: if true; }` rule.
- **Causal chain:** items list query → server rule check → no matching rule for `/items/` → default deny → `firestore/permission-denied`.
- **Proximate cause:** GitHub Actions deploy workflow does not deploy Firestore Rules, only the Cloudflare Pages SPA bundle. No human ran `firebase deploy --only firestore:rules` after PR #7 (PRJ-805) merged.
- **Root cause:** Missing CI step for Firestore Rules deployment; no automated assertion that deployed rules match the repo.

### Three-Element Proof

**1. Reproduction:** User screenshot shows `Could not load items: Missing or insufficient permissions. (firestore/permission-denied)` when entering folder `/folders/WzvKtUc9Ln1JtLSkGxm7` (PRJ-787 live deploy smoke test).

**2. Isolation:** REST API fetch of deployed ruleset `36eebe78-6f11-4939-96b8-40d4f9f2e4fe` returned the legacy prototype rules:
```
match /fabrics/{fabricId} { allow read, write: if true; }
match /folders/{folderId} { allow read, write: if true; }
match /purchaseOrders/{poId} { allow read, write: if true; }
match /activityLog/{logId} { allow read, write: if true; }
match /{path=**}/movements/{movementId} { allow read, write: if true; }
```
No `/items/` match, no catch-all. Firestore defaults to deny for unmatched paths. The repo `firestore.rules` (HEAD `a79c59e`) contains the full PRJ-805+873/874/865/866 ruleset that has NEVER been pushed. Last deployed ruleset updateTime is `2026-04-18T21:11:25.974643Z` — predates PR #7 (PRJ-805) merging on `2026-04-XX` and all subsequent rules changes.

**3. Completeness:** Explains every observation in IS/IS NOT:
- Folders list works → legacy `match /folders/{folderId} { allow read, write: if true; }` rule still active in deployed ruleset.
- Items list fails → no `/items/` rule deployed, default deny.
- AuthBar shows "Shaaiz" → falls back to `authUser.displayName` from Firebase Auth (line 98 in AuthBar.tsx fallback chain), since `/users/{uid}` Firestore read is ALSO denied (no `/users/` rule deployed).
- `isActive: true` is correct in Firestore → not the cause; the rule that would check it isn't deployed.
- UID matches → not the cause; the rule that would check existence isn't deployed.
- No recent code changes — repo is correct, deploy never happened.

H-001 (isActive type), H-002 (UID mismatch), H-003 (post-seed change), H-004 (composite index): all FALSIFIED by this root cause.
